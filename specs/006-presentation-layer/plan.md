## Summary

Build the complete v1 SwiftUI presentation layer over the merged Phase 1–4 stack, replacing the
diagnostic `ContentView` in `FinanceWorkspaceApp` with the real app. Strictly read-only: views render
engine projections; the only "actions" are navigation, selection, dry-run repair previews, and
Finder/editor reveals.

- **Design-token layer** (`DesignSystem/`): translate the `DESIGN.md` front matter into SwiftUI
  `Color`/`Font`/metric constants (light + dark) — the one allowed token-to-code translation point
  (per `swiftui-view-scaffold` preconditions). Built first; every view consumes it.
- **App shell** (`UI/Shell/`): rewritten `FinanceWorkspaceApp` scene + `.commands` menu set (§17),
  expanded `AppState` (@Observable root: workspace, indexing, module selection, navigation path,
  detail-pane state; Overview default), new `AppRouter` (typed `Route`, `NSUserActivity`
  encode/decode, programmatic KPI navigation), `NavigationSidebarView` (expandable groups, nested
  entity links, no Overview row, keyboard nav), global header (issues chip + sync chip, local-actions
  row, breadcrumb), `DetailPaneView` via the native `.inspector` slide-over (closed by default,
  six surface types).
- **Projection store** (`ProjectionStore`): the in-memory, non-authoritative read-model cache
  (technical-design §18) — parses the workspace off the main actor, runs all nine engines, and
  atomically swaps a `WorkspaceProjections` snapshot into `AppState` (FR-036); repair previews call
  the existing `RepairService` dry-run.
- **Shared components** (`UI/Shared/`): `KPICardView`, `PanelView`, `DataTableView`, `StatusChip`,
  `PeriodSelectorView`, `EmptyStateView`, `LoadingSkeletonView`, `SourceInspectorView`,
  `ValueProvenanceLabel`; charts (`PieChartView`, `SparklineView`, `BarChartView`) on **Swift
  Charts**; `HeatMapTableView` as a Grid-based color-scaled table sharing the chart-styling scale
  (research.md D4).
- **Module views**: `UI/Overview/` (5 live KPI cards, MoM panel, issues table + Preview Repair),
  `UI/Accounts/` (grid → group screen with inline ledger + business P&L → per-account screen;
  grouped multi-entry rows), `UI/Budget/` (pie + 50/50 panels + category table + drill-down;
  history; categories), `UI/SavingsInvestments/` (goals flat list + detail; holdings-focal portfolio
  with heat-map toggle, donut, sleeve table; holding detail with FIFO lots), `UI/Taxes/` (current
  year, prep checklist, read-only archive).
- **App target packaging** (US8, last): checked-in entitlements + `project.yml`; the `.xcodeproj` is
  generated by XcodeGen and built unsigned in a new macOS-CI step — resolving OOS-1/T004 without
  requiring local Xcode (research.md D2).

Technical approach: everything except the app-target wrapper stays inside the existing SwiftPM
`FinanceWorkspaceApp` executable target (SwiftUI + Swift Charts are SDK frameworks, CLT-buildable).
Views follow the five-layer rule — File → Parsing → Domain → Projection → **Presentation** — and
never compute finance logic; every view change clears the `design-adherence` gate.

## Technical Context

**Language/Version**: Swift 6; SwiftUI (`@Observable`, `NavigationSplitView`, `.inspector`,
`.commands`) and Swift Charts — all macOS-15-SDK system frameworks, no third-party dependencies.
**Primary Dependencies**: the merged `FinanceWorkspaceKit`: `WorkspaceManager`/providers
(`LocalFolderProvider` DEBUG, `ICloudContainerService` release — already wired in `AppState`),
`WorkspaceParser`/`WorkspaceContext`, `ValidationEngine` (issues chip/table), `RepairService`
(dry-run previews), `SettingsStore`, and the nine domain engines via `LinkingEngine`/
`OverviewEngine.dashboard(_:asOf:)`. Projections already carry `source_file`/`source_row`
provenance for the inspector. Packaging only: XcodeGen (brew, CI + optional local; the generated
`.xcodeproj` is not checked in).
**Storage**: none added. Views are read-only over canonical files; `ProjectionStore` is an
in-memory, regenerable cache (never authoritative — constitution P-I/II); session state
(`NSUserActivity` restoration payload) is OS-managed, not a file.
**Testing**: Swift Testing in macOS CI — unit tests for `AppRouter` route/`NSUserActivity`
round-trip, `ProjectionStore` atomic swap + read-only guarantee (byte-identical workspace,
SC-005), per-module view-model mapping (projection → component contract, typed states), menu
command enable/disable matrix. View rendering is verified via `#Preview` (light/dark) plus the
Milestone-5 manual demo script in `docs/test-plans.md`; XCUITest automation is deferred to
Phase 7. SwiftLint `--strict` on the Linux runner.
**Target Platform**: macOS 15 (Sequoia)+; min window 900 px per `DESIGN.md`.
**Project Type**: Native macOS desktop app — SwiftPM package (`FinanceWorkspaceKit` +
`FinanceWorkspaceApp` executable) plus a Phase-5 XcodeGen-generated app wrapper for
entitlement/packaging (US8).
**Performance Goals**: no visible stalls navigating a realistic 12-month fixture workspace;
projection build off the main actor with loading skeletons covering the gap; UI interactive during
full re-index (SC-010). Hard numeric thresholds and cache persistence are Phase 7 by roadmap.
**Constraints**: strictly zero workspace writes (SC-005; repair previews dry-run only; write
affordances visible-but-disabled); all visual values via the `DesignSystem` token layer from
`DESIGN.md` (light + dark); full keyboard navigability; typed engine states rendered as designed
states, never zero-filled; session-only selector persistence; `NSUserActivity`-dictionary deep
links; CLT-only dev box must build every view via `swift build` (Xcode work CI-gated, staged
last).
**Observability**: validation issues stream into the header chip + Overview table; sync/indexing
state into the sync chip; `os.Logger` (`Diagnostics`) for shell lifecycle events. No new logging
subsystem.
**Scale/Scope**: single user/workspace; ~1 shell (5 pieces) + token layer + ~12 shared components
+ 15 module views across 5 modules + router/store + app-target wrapper; 36 FRs, 8 user stories.

No open `NEEDS CLARIFICATION` — the four material ambiguities were resolved in the 2026-07-02
clarify session; the remaining plan-level choices (menu shortcut table, heat-map rendering, app
packaging mechanics, inspector implementation) are decided in `research.md` D1–D8.
